# Database And Data Model

## Current Database Status

There is no implemented database.

No Supabase client, migrations, SQL schema, generated database types, seed files, storage bucket definitions, RLS policies, triggers, functions, RPC calls, or backend queries exist in the current repo.

The data model is implemented as TypeScript interfaces in `src/types.ts` and in-memory objects generated by `src/services/courseGenerator.ts`.

## TypeScript Data Model

### `CourseProject`

Purpose: Top-level course project object used everywhere in the app.

Defined in `src/types.ts`.

Important fields:

- `id`
- `title`
- `description`
- `prompt`
- `settings`
- `theme`
- `status`
- `updatedAt`
- `outcomes`
- `modules`
- `pages`
- `assignments`
- `discussions`
- `quizzes`
- `rubrics`
- `assignmentGroups`
- `contactHours`
- `exportHistory`

Persistence: In-memory only.

### `CourseSettings`

Purpose: Intake and generation settings.

Important fields:

- `title`
- `description`
- `level`
- `modality`
- `creditHours`
- `lengthWeeks`
- `organizationPattern`
- `moduleCount`
- `themeId`
- `tone`
- `assignmentTypes`
- `quizFrequency`
- `discussionFrequency`
- `finalProject`
- `includeRubrics`
- `includeObjectives`
- `includeBloom`
- `includeContactHours`
- `accessibilityFocus`
- `sourceFiles`

Concern: Several settings are captured but not strongly enforced by generation. For example, generated course content always includes outcomes/contact-hour structures because the generator is not deeply conditional.

### `SourceFile`

Purpose: Attached file metadata.

Fields:

- `id`
- `name`
- `sizeLabel`
- `status`

Concern: There is no file content, MIME type, storage path, parsed text, checksum, upload status, or error state.

### `Theme`

Purpose: UI/export theme selection.

Fields:

- `id`
- `name`
- `accent`
- `accentDark`
- `soft`
- `contrastText`
- `bannerLabel`

Themes are defined in `src/data/themes.ts`.

### `CourseOutcome`

Purpose: Course-level learning outcomes.

Fields:

- `id`
- `code`
- `text`
- `bloomLevel`
- `alignedModuleIds`

Generated by `src/services/courseGenerator.ts`.

### `CourseModule` And `ModuleItem`

Purpose: Module sequence and Canvas module item references.

`CourseModule` fields:

- `id`
- `title`
- `description`
- `objectives`
- `workloadHours`
- `order`
- `expanded`
- `items`
- `status`

`ModuleItem` fields:

- `id`
- `type`
- `title`
- `refId`
- `order`
- `status`

Concern: Module items reference pages/assignments/discussions/quizzes by `refId`. Moving items across modules does not update the referenced object's own `moduleId`.

### `CoursePage`

Purpose: Canvas wiki page-like object.

Fields:

- `id`
- `title`
- `slug`
- `bodyHtml`
- `moduleId`
- `frontPage`
- `status`

Concern: HTML is user-editable and previewed with `dangerouslySetInnerHTML` in `src/App.tsx`. Export validation checks script tags but not all unsafe HTML patterns.

### `Assignment`

Purpose: Canvas assignment-like object.

Fields:

- `id`
- `title`
- `descriptionHtml`
- `points`
- `estimatedHours`
- `submissionType`
- `moduleId`
- `assignmentGroupId`
- `rubricId`
- `alignedOutcomeIds`
- `status`

Concern: Editor only exposes title and body, not points, submission type, due date, availability, group, rubric, or outcomes.

### `Discussion`

Purpose: Canvas discussion-like object.

Fields:

- `id`
- `title`
- `promptHtml`
- `points`
- `moduleId`
- `rubricId`
- `alignedOutcomeIds`
- `status`

Concern: Editor only exposes title and prompt HTML.

### `Quiz` And `QuizQuestion`

Purpose: Canvas quiz-like object.

`Quiz` fields:

- `id`
- `title`
- `purpose`
- `moduleId`
- `points`
- `questions`
- `alignedOutcomeIds`
- `status`

`QuizQuestion` fields:

- `id`
- `type`
- `stem`
- `choices`
- `correctAnswer`

- `points`

Concern: Exported QTI currently writes stems and metadata only; it does not fully serialize choices, correct answers, response processing, or feedback.

### `Rubric` And `RubricCriterion`

Purpose: Rubric structure for assignments/discussions.

Fields include:

- rubric `id`
- `title`
- `criteria`
- `points`
- criterion title/description/levels

Concern: Editor only exposes rubric title, not criteria or point structures.

### `AssignmentGroup`

Purpose: Gradebook group/weight.

Fields:

- `id`
- `name`
- `weight`
- `dropLowest`

Readiness checks that weights total 100%.

### `ContactHourPlan`

Purpose: Estimated workload/contact-hour breakdown.

Fields:

- `instructionalTime`
- `readingMediaTime`
- `assignmentTime`
- `discussionTime`
- `quizStudyTime`

- `justification`

### `ExportHistoryItem`

Purpose: Local export history row.

Fields:

- `id`
- `exportedAt`
- `fileName`
- `validationScore`

Concern: In-memory only; not reliable as audit or billing history.

### `ReadinessReport` And `ExportValidationReport`

Purpose: Client-side quality and package validation feedback.

Defined in `src/types.ts`, produced by `src/services/readiness.ts` and `src/services/imsccExport.ts`.

## Row-Level Security Policies

None exist.

Future Supabase implementation will need at minimum:

- User can read/write only own projects.
- Organization admins can manage organization projects and members.
- Instructors/designers can access assigned projects.
- Export jobs are scoped to project ownership.
- Source files are scoped to owner/org/project.
- Subscription status is read-only from trusted billing sync.

## Tables Or Queries That May Fail

There are no database queries.

The highest future mismatch risk is that `docs/COURSEFORGE_MVP_PLAN.md` describes production entities and status flows that are not represented as deployed tables. A future Supabase implementation must map the TypeScript model intentionally instead of assuming a database already exists.

## Data Integrity Concerns

- No persistence or autosave; all work disappears on refresh.
- Project IDs derive from title, so generating the same title can replace prior local project state.
- Module item movement can desynchronize module membership metadata.
- Duplicate module copies item references rather than duplicating underlying content.
- User-editable HTML has limited validation.
- Export history is local state only.
- Source files are metadata-only and not attached to project generation in a meaningful way.
